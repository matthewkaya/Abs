## Real bug found (and fixed) — caveat #12 was silently broken

`helm template --set cerbos.enabled=true` printed:

```
coalesce.go:237: warning: skipped value for abs.cerbos.env: Not a table.
```

Root cause: `infra/helm/abs/values.yaml` declared

```yaml
cerbos:
  env:
    CERBOS_NO_TELEMETRY: "1"
```

but the cerbos subchart's `values.yaml` (`charts/cerbos/values.yaml:80`) defines
`env: []` — a list of `{name, value}` entries. Helm's coalesce step refuses to
merge a map into a list, so the entire `cerbos.env` map was **dropped** before
reaching the cerbos pod. Caveat #12 ("Cerbos telemetry off") was therefore never
actually enforced in production renders, despite appearing to be set in
`values.yaml`.

A second instance of the bug lived in `templates/backend-configmap.yaml:23`,
which did `.Values.cerbos.env.CERBOS_NO_TELEMETRY` (map access). That line was
also wrong on its own — `CERBOS_NO_TELEMETRY` is a cerbos-process knob, never
read by our backend (`grep -r CERBOS_NO_TELEMETRY` confirms only helm references
exist).

### Fix

- `values.yaml`: convert to list form `- name: CERBOS_NO_TELEMETRY / value: "1"`,
  with an explanatory comment so a future reader understands the constraint.
- `backend-configmap.yaml`: drop the (unread) backend-side surfacing; replaced
  with a comment recording the ownership.

### Verification post-fix

```
$ helm template … --set cerbos.enabled=true
(no warnings)
$ grep -B2 -A3 CERBOS_NO_TELEMETRY /tmp/helm-render-r76-cerbos.yaml
          env:
            - name: CERBOS_NO_TELEMETRY
              value: "1"
```

The env var now actually lands on the cerbos pod's container env.
